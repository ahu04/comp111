Overview

Your shell program will print a prompt and allow the user to type a command. Commands will always be a single line, although they may contain spaces (so you may have to use fgets() to read the input). There are two types of commands: built-in commands and programs to run. When a built-in command is entered, the shell itself performs the requested action. Otherwise the first thing on the command line will be treated as the name of a program, and the shell will create a new process and load that program to run.

The only built-in command you will implement in your shell is exit. When exit is typed on the command line, your shell program should terminate. Otherwise, it should create a process for the specified program, wait for that child process to finish, and then print the exit status value of the child process. Then it should return to the prompt so the user can enter a new command. Print the status of the child process after it finishes.

If the child process fails before calling execvp or if execvp itself fails, then the child process should print an error message and exit with status 127 (we‚Äôll choose that number because it‚Äôs unlikely to already be used as an exit status by another program). Note that the shell (parent process) should simply print the exit status of the child process, whatever it is, and the child process should actually exit with status 127 if execvp fails (or there is any other error in the child process before execvp). The shell (parent process) should not make a special case for these errors and hard code printing 127; the number 127 comes from the child process, not the parent (shell).

dave@vm-hw09:~$ ./shell

Notice that all lines output from the shell program (parent or child process) start with ‚Äújsh‚Äù, and all the other output is from the new program that is run. This makes it easy to tell which lines comes from which source. Also, the lines with "jsh$ " are the prompt (Notice the space after the ‚Äò$‚Äô) and the rest of the line is the user input. Lines that begin with ‚Äújsh‚Äù but no ‚Äò$‚Äô are other output from the shell program (parent or child process).

Also, note that the exit status of 2 from ls filedoesnotexist in this example comes from the ls program itself, not from any code that you will write.

Once you have the basic functionality working, add output redirection to your shell. If you‚Äôre not familiar with output redirection in Unix/Linux shells, this redirection tutorial should help. Pay particular attention to the ‚ÄúPiping‚Äù section. You are going to implement piping in your shell, but you do not need to implement redirection to/from files. (Although knowing how to redirect to/from files on the command line is still a useful skill when working on Linux/Unix/MacOS, so I encourage you to learn about that as well, even though it‚Äôs not needed for this assignment.)

So when someone types something like this at your shell prompt:

jsh$ ./a arg1 | ./b | ./c arg2 arg3

The standard output (stdout) from ./a should not be printed on the terminal, but should instead be sent to the standard input (stdin) of ./b (rather than ./b reading stdin from keyboard input), and the stdout of ./b should be sent to stdin of ./c. Note that ./a, ./b, and ./c all run concurrently. ./a does not need to finish before ./b starts running, and ./b will receive output from ./a even while both continue running.

Notice that there are spaces on either side of the | character, but there don‚Äôt have to be. Your shell should work equally well with commands like a | b and a|b (i.e., with or without spaces around the |).

Be sure you run all the processes in the pipeline concurrently, not one after the other. Consider the situation where the first program in the pipline produces continuous output; you want it to proceed through the pipeline and print the resulting output (if any) from the last program as the first program runs. If you run them one at a time, you won‚Äôt get any output from the last program until after the first program finishes, and there are some scenarios where that may take a long time, or even continue running until someone terminates the first process manually. Each of the processes should be a child of the shell process (not of each other).

When you print the program‚Äôs exit status, you only need to print the status of the last program in the pipeline (e.g., from ./c in the example above). Note that this is the last one in order they‚Äôre written in the command, not necessarily the last child process to terminate. (There is no guarantee they will exit in the same order they‚Äôre written!)

Even though you only need to print the exit status of the last one, it is still important to wait (or waitpid) on all of the child processes. When a process is terminated, the OS still keeps an entry for it in the process table (list of all processes) and merely changes it‚Äôs state to ‚Äúterminated‚Äù (as opposed to ready, running, or blocked). Such a process is called a ‚Äúzombie process‚Äù because it will no longer be scheduled to run, but it still exists in the OS state. Once a ‚Äúzombie process‚Äù has been waited upon, the OS will free its resources and remove it from the process table. So failing to wait on a process can lead to a resource leak in the operating system.

Implementation

Each command will be a single line of input but may contain spaces that separate the command line arguments to the program. Using fgets() is probably the best way to read the input (or getline() may also work), and then use strtok_r() (or strsep()) to split the string at the spaces. There may be other ways to do this as well, and you‚Äôre welcome to do it any different way that works for this assignment (and is valid in C). However, note that there are reasons some of the alternative approaches may not work for more general shells even if they do work in this assignment, whereas the approach I‚Äôm pointing you at should work more generally. So that choice is up to you.

You will then need to use the fork() and execvp() functions to create a new process running the requested program with the specified command line arguments. Recall what you learned about these functions in class and in the textbook, and remember that you can (and should!) refer to the man pages for them. You are required to use the execvp() version of exec for this assignment, and may not use any of the other exec functions. You also may not use the system() or popen() functions.

To be clear, any of the following things will result in a 0 grade on this assignment:

  • calling system() function
  • calling popen() function
  • failing to call execvp() function at least once
  • failing to call fork() at least once
  • failing to compile successfully when you submit

You will then need to use either the wait() or waitpid() function to be sure the program completes before you prompt the user for the next command. Print the status that the program returned after the program finished but before displaying the prompt again. (Recall that the status is the return value the program gave when returning from main(), or the value it gave when calling exit().)

Print a sensible error message whenever anything goes wrong, and then return to the prompt so the user can enter a new command (unless the nature of the error makes that impossible, in which case print the message and exit). Be sure to check every function you call for errors!

The following man pages will be helpful for this portion of the assignment:

man fork
man execvp
man 2 wait

And the following may be helpful for parsing the user input:

man fgets
man strtok

To implement the output redirection (piping), you will need the pipe function (so now you see why it‚Äôs called ‚Äúpiping‚Äù and the | character is called a ‚Äúpipe‚Äù). The following man pages will be helpful for this part:

man 2 pipe
man dup

Be sure you‚Äôre using the pipe to connect each process‚Äô stdout directly to the next one‚Äôs stdin. The shell itself (parent process) should never read or write the pipe. If you try to do it that way, you will make things waaaaay harder on yourself than necessary (because you‚Äôll basically be re-implementing what the pipe has already implemented for you in your shell code).

Note that a process will not read an ‚Äúend of file‚Äù from a pipe (and will instead block and wait for more data to arrive in the pipe) if any process has an open file descriptor to the write end of the pipe. (This is because, in principle, more data could be written to the pipe at any moment. You may know that it won‚Äôt be, but the OS doesn‚Äôt know - it‚Äôs not inspecting the code to understand what it will do.) So it‚Äôs important that the parent process, or any other process that has a file descriptor to a pipe‚Äôs write end but does not intend to actually write to it, must close the write end file descriptor.

Also, if you use dup2(), pay careful attention to which argument should be which file descriptor number. Be sure you understand what it is doing to each.

Suggestions

You may proceed developing in whatever order you like, but we suggest doing it in the following order, testing thoroughly after each step:

  1. Write a program that forks a child then waits for it to finish before exiting. Print the status of the child process.
  2. Add exec to the child to run some program that you just hard code. (Don‚Äôt take user input.)
  3. Add the user input and modify exec to run the program and arguments provided by the user.
  4. Add the exit command to make the shell program exit when the user types this command instead of a program to run. (At this point there is no loop yet, so exit would just be something you type to not run a program at all.)
  5. Add the loop, so that (unless the user gives the exit command) the shell will return to the prompt and get another program to run after the previous one finished
  6. Allow the user to give multiple programs to run separated by pipe (|) characters. The shell should fork and run each of them, waiting for all the child processes to finish (and printing the last one‚Äôs status) before looping back to the prompt. (Note that the output is not piped yet, so they just run concurrently, each printing its own separate output. This is equivalent to what the real shell does if you use & instead of |.)
  7. Set up a pipe between each of the multiple programs so that the standard output (stdout) of each is redirected to the standard input (stdin) of the next. (stdout of the last one should not be redirected; its output will still be printed on the terminal.)

Be sure to pace yourself; you have approximately two weeks to work on this assignment. Steps 6 and 7 above are probably at least half the work of the assignment, so we strongly recommend finishing steps 1-5 by the end of the first week.

Also remember that short, simple, elegant code is much better than long, complex code. Just because a task is complex does not necessarily mean that the code needs to be (or should be) long and complex. Look for opportunities to write general code has a few small checks for where something needs to be done differently, rather than write completely different code paths for different scenarios that nonetheless have more in common than they do differently. (e.g., The scenario with a single program and no pipes is just a special case of a command with pipes, where the number of pipes happens to be zero.)

Remember to write error handling each time you call a system function!
